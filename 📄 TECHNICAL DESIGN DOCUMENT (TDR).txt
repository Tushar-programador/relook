📄 TECHNICAL DESIGN DOCUMENT (TDR)
🏷️ System Name

FeedSpace Backend & Platform

🧠 1. SYSTEM OVERVIEW
Goal

Build a multi-tenant SaaS platform that:

Collects feedback (video/audio/text)
Stores and processes media
Provides dashboard + embed widgets
Scales to thousands of users
🏗️ 2. HIGH-LEVEL ARCHITECTURE
[ Client (React) ]
        ↓
[ API Server (Node.js + Express) ]
        ↓
[ MongoDB ]       [ Object Storage (R2/S3) ]
        ↓
[ CDN for media delivery ]
🧩 3. ARCHITECTURE STYLE
Chosen Pattern

Layered Architecture (Service-based modular design)

Why?
Easier to scale later into microservices
Clean separation of concerns
Faster MVP development
📦 4. BACKEND STRUCTURE
src/
 ├── controllers/     → request handlers
 ├── services/        → business logic
 ├── models/          → mongoose schemas
 ├── routes/          → API routes
 ├── middlewares/     → auth, validation
 ├── utils/           → helpers
 ├── config/          → env configs
 └── app.js
⚙️ 5. TECH STACK DECISIONS
Backend
Node.js + Express
Why not NestJS?
Faster iteration
Less boilerplate
You already prefer JS flexibility
Database
MongoDB (Mongoose)
Why MongoDB?
Flexible schema (feedback types vary)
Fast iteration for MVP
Good for document-based data (feedback/media)
Storage
Cloudflare R2 (preferred) / AWS S3
Decision Reason
R2 → lower cost (no egress fees)
S3 → fallback if needed
Frontend
React + shadcn/ui + Tailwind

Feedback Collection
{
  "_id": "ObjectId",
  "projectId": "ObjectId",
  "type": "video|audio|text",
  "mediaUrl": "string",
  "thumbnailUrl": "string",
  "message": "string",
  "status": "pending|approved|rejected",
  "createdAt": "Date"
}
Indexes

index on status
compound: (projectId + status)

📤 9. MEDIA UPLOAD DESIGN
Approach

Direct Upload via Signed URL

Flow
Client → Backend → get signed URL  
Client → Upload to R2/S3  
Client → Send metadata to backend
Why?
Reduces backend load
Scales easily
